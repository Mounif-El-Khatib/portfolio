SemicolonCTF 2025 — Full Competition Writeup

SemicolonCTF 2025 × Toters 3 hours on-site 4th / 276 teams

SemicolonCTF 2025 was sponsored by Toters and had two rounds: a semifinal and a final. We placed 4th out of 276 teams in the final. There were a lot more challenges than what's here; each category had multiple problems across both rounds. These are the ones I worked on and remember well enough to write up properly. I didn't have time to screenshot anything during the competition, so this is reconstructed from memory and old chat logs.

01 — Crypto: AES-ECB Image Recovery (partial solve)

We got an encrypted image (catch_me.bmp) and a text file (catch_me.txt) with four lines: 

U2VtaWNvbG9uQ1RGMjV4VG90ZXJz                  — Line 1
77b7e24bb3642a4b9d3081d393785273              — Line 2
7dddbfabef0e23edd753c1006c1cbf3f99380a57fa    — Line 3
e94fd5250dcca0a3b0cea1651f0a821b              — Line 4

The challenge stated: Line 1 is "a clue in disguise", Line 2 is raw hex, Line 3 is a transformation of Line 2, and Line 4 "might unlock something vital". The image was encrypted with AES-128 in ECB mode.

What I figured out

Line 1 was base64. Decoding it gives SemicolonCTF25xToters, just the competition branding.

Line 2 is a 32-char hex string, so 128 bits, which is the correct size for an AES-128 key.

Line 3 is a 42-char hex string. The challenge said it's "a transformation of Line 2" but I wasn't able to figure out the exact relationship. It's not a standard hash length, because SHA-1 would be 40 chars, SHA-256 would be 64. Could be a truncated HMAC, some custom encoding, or something else entirely.

Line 4 is another 32-char hex string. Another AES-128-sized value.

Where I got stuck

I identified what each line likely was: base64, hex values, and a transformation I couldn't figure out, but I couldn't piece together how they all connected to get the actual decryption key. I tried using both Line 2 and Line 4 as the AES key but nothing produced a valid BMP header in the output.

The base64 decode on Line 1 was straightforward, and Lines 2 and 4 being AES-128 key length was a reasonable guess. But I never cracked the relationship between Lines 2 and 3, and without that I couldn't be sure which value was the actual decryption key.

02 — Android RE: APK Flag Hunting

We were given ctf_challenge.apk with flags buried in native libs, resources, and bytecode, along with a bunch of decoy flags meant to waste our time.

First pass

I started simple: strings piped into grep: 

strings ctf_challenge.apk | grep -i "CTF{"

Got a lot of matches back. I had to figure out the real ones from the fakes.

Sorting real from fake

I grepped through the decompiled output to trace where each flag actually came from: 

CTF{this_is_not_the_flag}             — decoy, showed up everywhere
CTF{adb_hunt3r_f0und_n4t1v3}          — native libraries (all 4 ABIs)
CTF{r3s0urc3_hunt3r_2024}             — resources.arsc
CTF{md5_XXXXXXXX}                     — format hint in classes.dex

The native libs had the same flag across all four architectures, which makes sense, same source code compiled for different ABIs: 

lib/arm64-v8a/libnative-lib.so
lib/x86/libnative-lib.so
lib/armeabi-v7a/libnative-lib.so
lib/x86_64/libnative-lib.so
CTF{this_is_not_the_flag} was everywhere. It flooded my grep output so I couldn't blindly pipe my way to the answer, I had to actually look at where each string lives and decide if it's real.

03 — Binary RE: Ghidra Native Library

This one involved a native library with an intentionally ugly name: za23jhfbsdveq, loaded by an activity called Tryhard. The Java side was bare bones: 

public class Tryhard extends AppCompatActivity {
    static { System.loadLibrary("za23jhfbsdveq"); }
    public native String ehjfbidphfaw();
}

This just loads the lib and calls a native method with a gibberish name. The actual logic was in the compiled native code.

Looking at it in Ghidra

I loaded the .so into Ghidra and checked the .rodata section. Found this sitting in memory: 

0001e8f8  43 54 46 7b 67 68 31 64 72 34   CTF{gh1dr4_g0nn4_h3lp_y0
          Pure virtual function called!

Looks like the flag bleeds into some C++ runtime garbage. If I just trusted strings output, the flag would be CTF{gh1dr4_g0nn4_h3lp_y0Pure virtual function called! which is obviously wrong.

The decompiled function allowed me to understand what was happening. It builds the string in three loops, each grabbing 8 characters from that data at sequential offsets: 

Loop 1 (offset 0x00): CTF{gh1d
Loop 2 (offset 0x08): r4_g0nn4
Loop 3 (offset 0x10): _h3lp_y0
Then tacks on:         ! and }

"Pure virtual function called!" just happened to be sitting right after the flag data in the read-only section. It's just a standard C++ string.

04 — PowerShell: Obfuscated Decryption Script

We were given a PowerShell script full of random stuff designed to distract us.

The noise

$obf = "f2OERT2bXUhzUkKSVI1>"
$this = "ignore_this_part_if_you_are_completely..."
$it = @(10, 20, 30, 40) | ForEach-Object { $_ * 3 }
Write-Host "Analyzing encrypted telemetry..."

Random variables that don't get used, arithmetic that goes nowhere and fake console output.

The actual payload

The real function in the script was an AES decryption routine:

function hQ1pZ($d) {
    $k = $env:CHALLENGE_KEY
    $a = New-Object System.Security.Cryptography.AesManaged
    $a.Mode = [System.Security.Cryptography.CipherMode]::ECB
    $a.Padding = [System.Security.Cryptography.PaddingMode]::PKCS7
    $a.Key = [System.Text.Encoding]::UTF8.GetBytes($k)
    $dc = $a.CreateDecryptor()
    $r = uJ3z1 $d      # base64 decode
    $o = $dc.TransformFinalBlock($r, 0, $r.Length)
    return [System.Text.Encoding]::UTF8.GetString($o)
}
$encData = "EsBJfwiG7Veox9lSNJjYRQ=="
Write-Host ("Output: " + (hQ1pZ $encData))

It base64 decodes a string and decrypts it with AES-ECB. The key comes from $env:CHALLENGE_KEY. There was also this function in the script: 

function eXa9d3B { return "ThisisIT" }

After identifying the decryption function and working out the key, running the script produced the flag. The hard part wasn't the crypto itself, it was recognizing which parts of the script mattered and which were there to waste time.

05 — Binaries: Pulling Flags with strings

We were also given a handful of compiled binaries. The flags were just embedded as plaintext string literals inside the binary data.

Finding this section's flags was really simple:

strings compiled_binary | grep -i "CTF{"

strings pulls out anything that looks like readable text from a binary file by default. Piped that into grep looking for the flag format and I was done.

The trick was not overthinking it. I could've loaded these into Ghidra or another disassembler, but that would've been overkill. strings and grep got the job quickly.

06 — AI Chatbot: Prompt Injection

We were given access to an AI chatbot with a flag somewhere in its configuration.

What first came to mind is trying prompt engineering. I tried to get the model to slip up and reveal stuff it was supposed to keep secret. Looking at the response, nothing useful appeared. So I went a little deeper and tried the same thing, but this time I monitored the responses by intercepting them with Burp Suite. The flag was right there in the raw JSON response.

The chatbot's only "security" was a system prompt telling it not to reveal the flag. The actual API had no restrictions, the flag was in every response, the frontend just didn't display it.

07 — Web

In the semi-final I got a flag by exploiting IDOR vulnerabilities. The app had a mailing list page that displayed user IDs right in the UI: 

ID: 1  — Alice
ID: 2  — Bob
ID: 3  — Charlie

The sign-out request looked something like:

POST /api/class/signout
{
  "user_id": 1,
  "class_id": 42
}

I grabbed a user's ID from the mailing list, intercepted the sign-out request with Burp, swapped in their ID, and the server just processed it. Their spot freed up, I registered, and found the flag.

08 — Team: OSINT & Machine Exploitation

Other team members handled the OSINT challenges and the vulnerable machine exploitation. Not writing those up since I wasn't the one working them, but they contributed to our overall placement.

Tools

  • strings + grep: first thing I ran on pretty much everything, quick way to pull readable text out of binaries
  • jadx: APK decompilation, Java source and dex bytecode inspection
  • Ghidra: native library disassembly and decompilation
  • Burp Suite: intercepting HTTP traffic, reading raw API responses the frontend didn't show
  • CyberChef: encoding/decoding chains, quick crypto operations
  • CrackStation: online hash lookup
  • base64 -d: command line base64 decoding

What I took away from it

The biggest factor in placing 4th wasn't technical skill, it was time management. I tried some basic web attacks like IDOR and SQL injection early on, but most of the web challenges needed more thorough work that would've come at the expense of easier points elsewhere. Same with crypto: lots of trial and error and guessing versus something like reverse engineering where I can just read the decompiled code and get a clear answer. The PowerShell script was similar: running it didn't give much, so I had to read it, ignore the noise, and find the one function that was relevant. Knowing when to move on was worth more than solving some of the challenges. The sunk cost fallacy doesn't care about your scoreboard.
